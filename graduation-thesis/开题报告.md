## 一、选题背景

### 1.1 研究现状综述

#### 1.1.1 即时通讯技术的发展

随着移动互联网的快速发展和智能终端的普及，即时通讯（Instant Messaging, IM）已成为人们日常生活和工作中不可或缺的一部分。根据中国互联网络信息中心（CNNIC）发布的第52次《中国互联网络发展状况统计报告》显示，截至2023年6月，我国即时通信用户规模达10.47亿，占网民整体的97.1%。

即时通讯技术经历了从简单的文本聊天到支持多媒体、语音视频通话的演进过程。早期的即时通讯系统如ICQ、MSN采用客户端-服务器架构，主要基于TCP长连接实现。随着Web技术的发展，基于HTTP的即时通讯方案逐渐出现，但传统的HTTP短轮询和长轮询方式存在延迟高、资源消耗大等问题。

WebSocket协议（RFC 6455）作为HTML5标准的一部分，于2011年被IETF标准化，为Web应用提供了全双工、低延迟的实时通信能力。相比传统HTTP方案，WebSocket具有以下显著优势：
- 全双工通信，服务器可主动推送消息
- 连接建立后数据帧头部仅2-10字节，大幅降低带宽消耗
- 消息推送延迟可降至毫秒级
- 持久连接，避免频繁的连接建立和断开

#### 1.1.2 分布式架构的必要性

随着用户规模的增长，单体架构的即时通讯系统面临性能瓶颈、单点故障风险和扩展性限制等问题。分布式架构通过服务拆分、负载均衡、数据分片等技术手段，可以实现：
- **水平扩展**：通过增加服务器节点提升系统处理能力
- **高可用性**：避免单点故障，提升系统可靠性
- **服务解耦**：不同功能模块独立部署和升级
- **资源优化**：根据业务负载动态调整资源分配

```

应用层（Spring Boot）

    ├── 好友模块
    ├── 群组模块
    ├── 消息模块
    ├── 文件模块
    └── 连接模块
    ↓

    ├── MySQL（关系型数据）
    ├── MongoDB（消息存储）
```

#### 2.2.3 核心技术实现

**1. WebSocket连接管理**
- 基于Spring WebSocket实现长连接
- 使用STOMP协议进行消息路由
- 实现心跳检测机制（30秒间隔）
- 支持断线重连（指数退避算法）

**2. 消息可靠性保障**
- 消息ACK机制：三态确认（发送、送达、已读）
- 离线消息：存储到MongoDB，上线后推送
- 消息重试：失败消息自动重试3次
- 消息去重：基于雪花算法生成的消息ID去重

**3. 混合存储方案**
- MySQL：存储用户信息、好友关系、群组信息
- MongoDB：存储聊天消息、离线消息
- Redis：缓存用户在线状态、会话列表、热点数据

**4. 多级缓存策略**
- L1缓存：本地缓存（Caffeine），TTL 5分钟
- L2缓存：Redis分布式缓存，TTL 30分钟
- 缓存更新：写操作同步更新缓存
- 缓存预热：系统启动时加载常用数据

**5. 分布式ID生成**
- 采用雪花算法（Snowflake）
- 保证ID全局唯一且递增
- 支持多机房部署

### 3.3 拟解决的主要问题及措施

#### 问题1：WebSocket连接的稳定性

**问题描述**：
移动网络环境下，网络不稳定导致WebSocket连接频繁断开，影响用户体验。

**解决措施**：
1. 实现心跳检测机制，每30秒发送一次心跳包，及时发现连接异常
2. 实现断线重连机制，采用指数退避算法（1s、2s、4s、8s）
3. 客户端维护连接状态，自动重连并恢复会话
4. 服务端保存连接上下文，重连后快速恢复

#### 问题2：分布式环境下的消息可靠性

**问题描述**：
在分布式部署环境下，如何保证消息不丢失、不重复、有序送达。

**解决措施**：
1. **消息ACK机制**：实现三态确认（已发送、已送达、已读），确保消息送达
2. **离线消息处理**：用户离线时消息存储到MongoDB，上线后推送
3. **消息重试机制**：发送失败的消息自动重试3次，重试失败进入死信队列
4. **消息去重**：基于雪花算法生成全局唯一消息ID，使用Redis记录已处理消息
5. **消息顺序保证**：单聊消息基于消息ID递增性保证顺序，群聊消息使用消息队列的顺序消费

#### 问题3：高并发场景下的性能瓶颈

**问题描述**：
在高并发场景下，数据库读写压力大，系统响应慢，如何提升性能。

**解决措施**：
1. **多级缓存**：本地缓存（Caffeine）+ Redis分布式缓存，减少数据库访问
2. **数据库优化**：
   - 合理设计索引，优化查询语句
   - MySQL读写分离，读操作分流到从库
   - MongoDB使用分片集群，按会话ID分片
3. **异步处理**：使用RocketMQ消息队列，将消息发送异步化
4. **连接池优化**：合理配置数据库连接池和Redis连接池参数
5. **接口限流**：使用OpenResty + Lua实现接口限流，防止恶意请求

#### 问题4：海量消息的存储与查询

**问题描述**：
即时通讯系统每天产生海量消息，如何高效存储和快速查询。

**解决措施**：
1. **混合存储**：
   - MySQL存储结构化数据（用户、好友、群组）
   - MongoDB存储非结构化数据（聊天消息）
2. **数据冷热分离**：
   - 热数据（近7天）：MongoDB主集群，SSD存储
   - 温数据（7-90天）：MongoDB从集群
   - 冷数据（90天以上）：归档到对象存储（OSS）
3. **索引优化**：
   - MongoDB创建复合索引（会话ID + 时间戳）
   - 定期清理过期索引
4. **分页查询**：历史消息采用分页加载，避免一次性加载大量数据

#### 问题5：系统的可扩展性和高可用性

**问题描述**：
如何设计系统架构，使其能够水平扩展，并保证高可用性。

**解决措施**：
1. **微服务架构**：按功能模块拆分服务，独立部署和扩展
2. **负载均衡**：使用OpenResty实现负载均衡，将请求分发到多个后端节点
3. **服务注册与发现**：使用Nacos实现服务注册与发现（可选）
4. **容器化部署**：使用Docker容器化部署，便于快速扩容
5. **数据库高可用**：
   - MySQL主从复制，主库故障时自动切换到从库
   - MongoDB副本集，保证数据高可用
   - Redis哨兵模式，实现自动故障转移
6. **降级策略**：关键服务故障时，启用降级方案，保证核心功能可用
